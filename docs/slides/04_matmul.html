Step 0: What GEMM Does

A (M x K)
x
B (K x N)
→
C (M x N)

Each output cell C[row, col] is a dot product. The whole optimization story is really about reusing the same A and B values many times.

Step 1: Naive GEMM

float acc = 0.0f;
for (int inner = 0; inner < k; ++inner) {
  acc += a[row * k + inner] * b[inner * n + col];
}
c[row * n + col] = acc;
Good Very easy to understand.
Bad Different threads reload the same A and B values again and again from global memory.

Why Naive GEMM Hits A Wall

Repeated global loads
A
B
The same inputs are useful to many outputs, but the naive kernel does not exploit that reuse.
The next idea
Key insight Load a tile once, then reuse it many times from shared memory.

Step 2: Shared-Memory Tiled GEMM

One block computes one C tile
A tile
x
B tile
→
C tile
for (int tile_k = 0; tile_k < k; tile_k += kTile) {
  a_tile[ty][tx] = a[row * k + tile_k + tx];
  b_tile[ty][tx] = b[(tile_k + ty) * n + col];
  __syncthreads();

  for (int inner = 0; inner < kTile; ++inner) {
    acc += a_tile[ty][inner] * b_tile[inner][tx];
  }
  __syncthreads();
}
Shared memory is the first big leap: load once, reuse many times.

Why Another Step Is Needed

Shared memory helped But each thread still computes only a small amount of output.
Next question Can each thread and warp hold more useful work locally?

Step 3: Warp And Register Tiling

Hierarchy inside one block
Block tile Owned by the whole block.
Warp tile Owned by one warp.
Thread tile Owned by one thread.
Registers Hold the partial output tile.
float acc[2][2] = {0.0f};
acc[0][0] += a_frag0 * b_frag0;
acc[0][1] += a_frag0 * b_frag1;
acc[1][0] += a_frag1 * b_frag0;
acc[1][1] += a_frag1 * b_frag1;
One thread now accumulates multiple outputs, which increases local reuse.

Step 4: Pipeline-Oriented Thinking

load next tile
||
compute current tile

The latest version starts to look like a modern kernel: not just reuse, but overlap. While one tile is being consumed, the next tile is being prepared.
